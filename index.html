architecture

One process vs. five

The core trade-off being tested: keep everything in one address space for speed, or split it into specialized services for resilience.

monolithic

Weaviate

Single Go binary. Vector index (HNSW), storage, and request handling share one memory space — no serialization cost between components, but no isolation either. A memory spike anywhere takes the whole process down.

microservices

Milvus

Proxy, coordinator, index nodes and query nodes run as separate processes talking over gRPC, with storage (segments) offloaded to MinIO. More moving parts and a network hop per request, but no single point of memory failure.

result 01 — ingestion

Throughput vs. stability under write load

At 128 dimensions, Milvus actually got faster as the dataset grew — startup overhead from etcd/MinIO coordination gets amortized over more vectors. Weaviate did the opposite: every new vector makes the HNSW graph more expensive to update, and write speed degrades as the dataset scales.

Milvus Weaviate OOM crash — no data point
Milvus: Small 13700 VPS, Big 26500 VPS. Weaviate: Small 427 VPS, Medium 299 VPS, Big: OOM crash.

At the Big setting, Weaviate didn't just slow down — it didn't finish. Every "Big" ingestion run across all three dimensions ended in an OOM kill. Milvus completed all nine.

result 02 — dimensionality

Higher dimensions cost everyone — but not equally

Moving from 128d to 1024d embeddings (typical of modern LLM outputs) makes every distance calculation linearly more expensive. Milvus throughput dropped ~5.9x; it stayed CPU-bound by flushing segments to disk instead of holding everything in RAM. Weaviate, which keeps its HNSW graph resident in memory, fell even further behind at every dimension tested.

Milvus Weaviate
Milvus at Medium size: 21566 VPS (128d), 9113 VPS (512d), 4472 VPS (1024d). Weaviate: 299, 204, 141 VPS.
result 03 — the "bigger-than-RAM" wall

What actually survived 8GB of RAM

Two separate questions: could each system ingest a dataset, and could it then query it? HNSW indexes need to live in RAM to be searched — so a successful ingest doesn't guarantee a queryable index once the data is loaded back for search.

result 03 — incident report

Anatomy of a crash

The Weaviate 512d-Big run, reconstructed from container logs and docker stats sampling:

[start] container weaviate_db begins ingesting the 512d-Big set
[ramp-up] HNSW graph grows, RAM climbs steadily — sawtooth CPU pattern (GC pressure)
[approach] RAM nears the container's 7.7GB hard limit
[repeat] client errors: ReadTimeout, then RemoteDisconnected
[end] Linux OOM killer invoked — process terminated

Milvus completed the same 512d-Big ingest without incident, with a flat, bounded memory profile — it offloads sealed segments to disk instead of fighting to keep the whole index resident in RAM.

result 04 — query performance

Once it's in memory, Weaviate is fast — really fast

Pure k-NN search (no filtering) tells the opposite story to ingestion. For datasets that fit comfortably in RAM, Weaviate's single-process design avoids Milvus's internal gRPC hop between proxy and query nodes — and it shows, especially in the tail.

result 05 — filtered search

Add a metadata filter, and the lead flips

Pure vector search favors Weaviate's tight, single-process loop. But combine a vector search with a boolean metadata filter (city_id == X AND quality_score > Y) and Milvus's bitmap pre-filtering takes over — it can skip ineligible vectors before the expensive distance calculation even starts.

Milvus: 4.9ms, 6.2ms, 11.7ms. Weaviate: 5.8ms, 11.3ms, 26.2ms.

The gap widens further at the tail, even at the Medium dataset size: Milvus's P95 stays at 6.9ms while Weaviate's climbs to 31.9ms. Milvus's bitmap masking isolates filter cost from the vector search itself; Weaviate has to reconcile its inverted index with the HNSW graph on every combined query.

conclusion

The sprinter and the marathon runner

Neither database "wins" outright — the results trace a clean trade-off between raw query speed and the ability to survive scale.

the sprinter

Weaviate

Near-zero jitter, up to 2x the query throughput, and excellent equality-filter performance — as long as the dataset stays inside the RAM budget. Push past that line and it doesn't degrade, it crashes.

the marathon runner

Milvus

A steady 5-10ms network tax on every query, in exchange for ingestion that gets faster with scale, 2.3x better storage efficiency, and zero crashes across all nine dataset configurations.
